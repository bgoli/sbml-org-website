XMLErrorLog_t Class Reference
Core libSBML

Detailed Description

Log of diagnostics reported during XML processing.

This class of objects is defined by libSBML only and has no direct equivalent in terms of SBML components. This class is not prescribed by the SBML specifications, although it is used to implement features defined in SBML.

The error log is a list. The XML layer of libSBML maintains an error log associated with a given XML document or data stream. When an operation results in an error, or when there is something wrong with the XML content, the problem is reported as an XMLError_t object stored in the XMLErrorLog_t list. Potential problems range from low-level issues (such as the inability to open a file) to XML syntax errors (such as mismatched tags or other problems).

A typical approach for using this error log is to first use getNumErrors() to inquire how many XMLError_t object instances it contains, and then to iterate over the list of objects one at a time using getError(unsigned int n) const. Indexing in the list begins at 0.

In normal circumstances, programs using libSBML will actually obtain an SBMLErrorLog_t rather than an XMLErrorLog_t. The former is subclassed from XMLErrorLog_t and simply wraps commands for working with SBMLError_t objects rather than the low-level XMLError_t objects. Classes such as SBMLDocument_t use the higher-level SBMLErrorLog_t.

LIBLAX_EXTERN void XMLErrorLog_setSeverityOverride ( XMLErrorLog_t *  log,
XMLErrorSeverityOverride_t  overridden 
)

Set the severity override of the given log to the given overridden value.

Parameters
logXMLErrorLog_t structure to be queried.
overriddenan override code indicating what to do. If the value is LIBSBML_OVERRIDE_DISABLED (the default setting) all errors logged will be given the severity specified in their usual definition. If the value is LIBSBML_OVERRIDE_WARNING, then all errors will be logged as warnings. If the value is LIBSBML_OVERRIDE_DONT_LOG, no error will be logged, regardless of their severity.
