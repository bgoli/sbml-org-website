126 * <h2>Consistency and adherence to SBML specifications</h2>
127 <p>
128 * To make it easier for applications to do whatever they need,
129 * libSBML version 5.18.0
130 * is relatively lax when it comes to enforcing correctness and
131 * completeness of models <em>during</em> model construction and editing.
132 * Essentially, libSBML <em>will</em> <em>not</em> in most cases check automatically
133 * that a model's components have valid attribute values, or that the
134 * overall model is consistent and free of errors&mdash;even obvious errors
135 * such as duplication of identifiers.  This allows applications great
136 * leeway in how they build their models, but it means that software
137 * authors must take deliberate steps to ensure that the model will be, in
138 * the end, valid SBML.  These steps include such things as keeping track
139 * of the identifiers used in a model, manually performing updates in
140 * certain situations where an entity is referenced in more than one place
141 * (e.g., a species that is referenced by multiple {@link SpeciesReference}
142 * objects), and so on.
143 <p>
144 * That said, libSBML does provide powerful features for deliberately
145 * performing validation of SBML when an application decides it is time to
146 * do so.  The interfaces to these facilities are on the {@link SBMLDocument}
147 * class, in the form of {@link SBMLDocument#checkInternalConsistency()} and
148 * {@link SBMLDocument#checkConsistency()}.  Please refer to the documentation for
149 * {@link SBMLDocument} for more information about this.
150 <p>
151 * While applications may play fast and loose and live like free spirits
152 * during the construction and editing of SBML models, they should always
153 * make sure to call {@link SBMLDocument#checkInternalConsistency()} and/or
154 * {@link SBMLDocument#checkConsistency()} before writing out the final version of
155 * an SBML model.

157 * <h2>Model attributes introduced in SBML Level&nbsp;3</h2>
158 <p>
159 * As mentioned above, the {@link Model} class has a number of optional attributes
160 * in SBML Level&nbsp;3.  These are 'substanceUnits',
161 * 'timeUnits', 'volumeUnits', 'areaUnits', 'lengthUnits', 'extentUnits',
162 * and 'conversionFactor.  The following provide more information about
163 * them.
164 <p>
165 * <h3>The 'substanceUnits' attribute</h3>
166 <p>
167 * The 'substanceUnits' attribute is used to specify the unit of
168 * measurement associated with substance quantities of {@link Species} objects that
169 * do not specify units explicitly.  If a given {@link Species} object definition
170 * does not specify its unit of substance quantity via the 'substanceUnits'
171 * attribute on the {@link Species} object instance, then that species inherits the
172 * value of the {@link Model} 'substanceUnits' attribute.  If the {@link Model} does not
173 * define a value for this attribute, then there is no unit to inherit, and
174 * all species that do not specify individual 'substanceUnits' attribute
175 * values then have <em>no</em> declared units for their quantities.  The
176 * SBML Level&nbsp;3 specifications provide more details.
177 <p>
178 * Note that when the identifier of a species appears in a model's
179 * mathematical expressions, the unit of measurement associated with that
180 * identifier is <em>not solely determined</em> by setting 'substanceUnits'
181 * on {@link Model} or {@link Species}.  Please see the discussion about units given in
182 * the documentation for the {@link Species} class.
183 <p>
184 * <h3>The 'timeUnits' attribute</h3>
185 <p>
186 * The 'timeUnits' attribute on SBML Level&nbsp;3's {@link Model} object is used to
187 * specify the unit in which time is measured in the model.  This attribute
188 * on {@link Model} is the <em>only</em> way to specify a unit for time in a model.
189 * It is a global attribute; time is measured in the model everywhere in
190 * the same way.  This is particularly relevant to {@link Reaction} and {@link RateRule}
191 * objects in a model: all {@link Reaction} and {@link RateRule} objects in SBML define
192 * per-time values, and the unit of time is given by the 'timeUnits'
193 * attribute on the {@link Model} object instance.  If the {@link Model} 'timeUnits'
194 * attribute has no value, it means that the unit of time is not defined
195 * for the model's reactions and rate rules.  Leaving it unspecified in an
196 * SBML model does not result in an invalid model in SBML Level&nbsp;3;
197 * however, as a matter of best practice, we strongly recommend that all
198 * models specify units of measurement for time.
199 <p>
200 * <h3>The 'volumeUnits', 'areaUnits', and 'lengthUnits' attributes</h3>
201 <p>
202 * The attributes 'volumeUnits', 'areaUnits' and 'lengthUnits' together are
203 * used to set the units of measurements for the sizes of {@link Compartment}
204 * objects in an SBML Level&nbsp;3 model when those objects do not
205 * otherwise specify units.  The three attributes correspond to the most
206 * common cases of compartment dimensions: 'volumeUnits' for compartments
207 * having a 'spatialDimensions' attribute value of <code>'3'</code>, 'areaUnits' for
208 * compartments having a 'spatialDimensions' attribute value of <code>'2'</code>, and
209 * 'lengthUnits' for compartments having a 'spatialDimensions' attribute
210 * value of <code>'1'.</code>  The attributes are not applicable to compartments
211 * whose 'spatialDimensions' attribute values are <em>not</em> one of <code>'1'</code>,
212 * <code>'2'</code> or <code>'3'.</code>
213 <p>
214 * If a given {@link Compartment} object instance does not provide a value for its
215 * 'units' attribute, then the unit of measurement of that compartment's
216 * size is inherited from the value specified by the {@link Model} 'volumeUnits',
217 * 'areaUnits' or 'lengthUnits' attribute, as appropriate based on the
218 * {@link Compartment} object's 'spatialDimensions' attribute value.  If the {@link Model}
219 * object does not define the relevant attribute, then there are no units
220 * to inherit, and all {@link Compartment} objects that do not set a value for
221 * their 'units' attribute then have <em>no</em> units associated with
222 * their compartment sizes.
223 <p>
224 * The use of three separate attributes is a carry-over from SBML
225 * Level&nbsp;2.  Note that it is entirely possible for a model to define a
226 * value for two or more of the attributes 'volumeUnits', 'areaUnits' and
227 * 'lengthUnits' simultaneously, because SBML models may contain
228 * compartments with different numbers of dimensions.
229 <p>
230 * <h3>The 'extentUnits' attribute</h3>
231 <p>
232 * Reactions are processes that occur over time.  These processes involve
233 * events of some sort, where a single ``reaction event'' is one in which
234 * some set of entities (known as reactants, products and modifiers in
235 * SBML) interact, once.  The <em>extent</em> of a reaction is a measure of
236 * how many times the reaction has occurred, while the time derivative of
237 * the extent gives the instantaneous rate at which the reaction is
238 * occurring.  Thus, what is colloquially referred to as the 'rate of the
239 * reaction' is in fact equal to the rate of change of reaction extent.
240 <p>
241 * In SBML Level&nbsp;3, the combination of 'extentUnits' and 'timeUnits'
242 * defines the units of kinetic laws in SBML and establishes how the
243 * numerical value of each {@link KineticLaw} object's mathematical formula is
244 * meant to be interpreted in a model.  The units of the kinetic laws are
245 * taken to be 'extentUnits' divided by 'timeUnits'.
246 <p>
247 * Note that this embodies an important principle in SBML Level&nbsp;3
248 * models: <em>all reactions in an SBML model must have the same units</em>
249 * for the rate of change of extent.  In other words, the units of all
250 * reaction rates in the model <em>must be the same</em>.  There is only
251 * one global value for 'extentUnits' and one global value for 'timeUnits'.
252 <p>
253 * <h3>The 'conversionFactor' attribute</h3>
254 <p>
255 * The attribute 'conversionFactor' in SBML Level&nbsp;3's {@link Model} object
256 * defines a global value inherited by all {@link Species} object instances that do
257 * not define separate values for their 'conversionFactor' attributes.  The
258 * value of this attribute must refer to a {@link Parameter} object instance
259 * defined in the model.  The {@link Parameter} object in question must be a
260 * constant; ie it must have its 'constant' attribute value set to
261 * <code>'true'.</code>
262 <p>
263 * If a given {@link Species} object definition does not specify a conversion
264 * factor via the 'conversionFactor' attribute on {@link Species}, then the species
265 * inherits the conversion factor specified by the {@link Model} 'conversionFactor'
266 * attribute.  If the {@link Model} does not define a value for this attribute,
267 * then there is no conversion factor to inherit.  More information about
268 * conversion factors is provided in the SBML Level&nbsp;3
269 * specifications.

1210   <p>
1211   * <p>
1212 * <p>
1213 * The identifier given by an object's 'id' attribute value
1214 * is used to identify the object within the SBML model definition.
1215 * Other objects can refer to the component using this identifier.  The
1216 * data type of 'id' is always <code>SId</code> or a type derived
1217 * from that, such as <code>UnitSId</code>, depending on the object in

1220 *   letter .= 'a'..'z','A'..'Z'
1221 *   digit  .= '0'..'9'
1222 *   idChar .= letter | digit | '_'
1223 *   SId    .= ( letter | '_' ) idChar*
1224 * </pre>

1226 * the character <code>*</code> 'zero or more times', and the character
1227 * <code>|</code> indicates logical 'or'.  The equality of SBML identifiers
1228 * is determined by an exact character sequence match; i.e., comparisons must
1229 * be performed in a case-sensitive manner.  This applies to all uses of
1230 * <code>SId</code>, <code>SIdRef</code>, and derived types.
1231 <p>
1232 * Users need to be aware of some important API issues that are the result of
1233 * the history of SBML and libSBML.  Prior to SBML Level&nbsp;3
1234 * Version&nbsp;2, SBML defined 'id' and 'name' attributes on only a subset
1235 * of SBML objects.  To simplify the work of programmers, libSBML's API

1237 * instead of on individual subobject classes. This made the
1238 * get/set/etc. methods uniformly available on all objects in the libSBML
1239 * API.  LibSBML simply returned empty strings or otherwise did not act when
1240 * the methods were applied to SBML objects that were not defined by the SBML
1241 * specification to have 'id' or 'name' attributes.  Additional complications
1242 * arose with the rule and assignment objects: {@link InitialAssignment},
1243 * {@link EventAssignment}, {@link AssignmentRule}, and {@link RateRule}.  In early versions of SBML,
1244 * the rule object hierarchy was different, and in addition, then as now,
1245 * they possess different attributes: 'variable' (for the rules and event
1246 * assignments), 'symbol' (for initial assignments), or neither (for
1247 * algebraic rules).  Prior to SBML Level&nbsp;3 Version&nbsp;2, getId()
1248 * would always return an empty string, and isSetId() would always return 
1249 * <code>false</code> for objects of these classes.

1251 * With the addition of 'id' and 'name' attributes on {@link SBase} in Level&nbsp;3
1252 * Version&nbsp;2, it became necessary to introduce a new way to interact
